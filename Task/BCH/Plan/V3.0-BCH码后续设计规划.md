# 基于公共基础工程的 BCH 码后续设计规划

结合你上传的公共基础工程交接说明和老师的方案文档，现在 BCH 设计已经不再是从零搭建一套独立仿真程序，而是：

> **在已经完成的 `Task/Common` 公共框架上，只实现 BCH 专用的码长适配、编码、译码、分块恢复和专用统计。**

公共工程已经提供公共类型、200/300 bit 帧池、按帧独立噪声、BPSK、AWGN、硬判决、BER/FER、时延、停止条件、checkpoint、shard、结果格式和绘图。BCH 不应再复制这些功能。fileciteturn4file0

下面给出从当前状态开始的 BCH 完整实施规划。

# 一、先纠正并冻结几个关键边界

## 1. 码率定义

整个 BCH 仿真统一使用：

\[
R=\frac{K_{\text{payload}}}{N_{\text{encoded}}}
\]

其中：

- \(K_{\text{payload}}\)：原始输入电文长度；
- \(N_{\text{encoded}}\)：经过 BCH 编码后实际进入 BPSK 的总码长。

对于四个正式 Case：

| Case | 原始输入 | 编码后码长 | 码率 |
|---|---:|---:|---:|
| BCH-S200 | 200 | 285 | \(200/285\approx0.701754\) |
| BCH-S300 | 300 | 420 | \(300/420\approx0.714286\) |
| BCH-B200 | 200 | 248 | \(200/248\approx0.806452\) |
| BCH-B300 | 300 | 390 | \(300/390\approx0.769231\) |

公共交接文件也明确要求：分子使用原始 payload，分母使用编码后实际进入信道的码长，不能把 filler 算进分子，也不能直接用单块母码的 \(k/n\) 替代整帧码率。fileciteturn4file0

---

## 2. 噪声规则

老师要求“每一帧噪声随机生成”，公共工程已经通过**按帧索引确定的独立标准高斯噪声**实现了这一点。

噪声样本由以下信息唯一确定：

```text id="hk5kcz"
masterNoiseSeed
noiseGroupId
frameIndex
symbolIndex
noisePolicyVersion
```

因此：

- frame 0 和 frame 1 的噪声不同；
- 同一 frame 在不同 SNR 下使用相同的标准高斯母样本，只改变 \(\sigma\)；
- 相同配置能够复现；
- 不依赖执行顺序、线程号或当前时间；
- 不存在所有帧重复使用同一个噪声向量的问题。fileciteturn4file0

BCH 只负责告诉公共噪声模块当前编码后码长 \(N\)，然后读取当前 frame 的前 \(N\) 个标准高斯样本。

---

## 3. 查表译码边界

根据老师的最新要求，必须冻结为：

```text id="5n1gjd"
BCH(15,11,1) 分块方案：
使用 syndrome 查表译码。

整块缩短 BCH：
使用 BM + Chien 代数译码。
```

上传的公共交接说明中有一句“分块模式至少实现代数译码和查表译码”，这是早期较宽的建议。结合老师后来明确提出的“查表只在分块的做”，本轮更合理的主线是：

- 分块主译码器：查表；
- 整块主译码器：BM + Chien；
- 不要求分块方案再额外实现一套 BM + Chien 作为正式性能 Case；
- 但可使用独立数学枚举或 MATLAB 对分块查表结果进行验证；
- 不把“分块查表与分块代数译码性能对比”列为必做实验。

这样既满足老师要求，也避免无意义地重复两套分块译码器。

# BCH-02：BCH(15,11,1) 单块编码器

分块方案先独立完成短码编码器，因为它参数明确、规模小、可穷举。

## 算法

\[
g(x)=x^4+x+1
\]

\[
r(x)=x^4m(x)\bmod g(x)
\]

\[
c(x)=x^4m(x)+r(x)
\]

## 流程

```text id="vs9wr8"
输入 11 bit
        ↓
按冻结位序映射 m(x)
        ↓
左移 4 位
        ↓
GF(2) 多项式除法
        ↓
得到 4 bit 余式
        ↓
拼接系统码
        ↓
输出 15 bit
```

## 测试

- 全零；
- 全一；
- 单高位；
- 单低位；
- 交替比特；
- 固定向量；
- 全部 \(2^{11}=2048\) 个输入。

必须满足：

```text id="q9xp83"
全部码字 c(x) mod g(x) = 0
系统信息位未改变
2048 个输入对应 2048 个唯一合法码字
```

## 初始 SNR

可暂用：

```text id="4ilsfp"
0, 2, 4, 6 dB
```

最终由运行结果调整。

## 帧数

建议固定读取前 100 帧，不提前停止。

## 流程

```text id="4uwcy0"
读取当前 frame
        ↓
BCH 编码
        ↓
公共 BPSK
        ↓
按 K/N 计算 sigma
        ↓
读取当前 frame 独立噪声前 N 项
        ↓
公共 AWGN
        ↓
公共硬判决
        ↓
BCH 译码
        ↓
恢复 payload
        ↓
公共 BER/FER
        ↓
BCH detail
        ↓
公共 CSV/PNG
```

## 检查

- 每个 Case 能运行；
- 不同 frame 噪声不同；
- 同一 frame 在不同 Case 的公共噪声前缀一致；
- BER/FER 总体随 SNR 下降；
- 无 NaN/Inf；
- 结果文件名明确；
- checkpoint 字段完整；
- 编码和译码时间边界正确。

# BCH-17：突发错误模型与交织

老师明确要求交织只在突发场景使用，因此单独放在后期。

## 突发错误模型

至少考虑：

```text id="e6bpwl"
连续硬比特翻转
连续符号擦除
连续强干扰
短时遮挡
```

必须冻结：

- 突发长度；
- 起点；
- 每帧次数；
- 是否跨 BCH 子块；
- 是否叠加 AWGN；
- 随机种子。

## 交织

```text id="lqbs7o"
无交织
块交织
行列交织
固定 seed 伪随机交织
```

必须先验证：

\[
\Pi^{-1}(\Pi(x))=x
\]

## 实验重点

分块 BCH 应重点分析：

```text id="yuv19u"
交织前：
一个突发集中进入少数 BCH(15,11,1) 块，
容易使单块错误数超过 1。

交织后：
错误被分散到多个 BCH 块，
可能使每块错误数降至 0 或 1。
```

## 输出

- FER 改善；
- successRate 改善；
- 误纠改善；
- 缓存长度；
- 交织时间；
- 去交织时间；
- 附加时延；
- 内存。
