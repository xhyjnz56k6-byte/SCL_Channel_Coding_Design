# 三类信道编码公共基础仿真详细规划

## 0. 文档说明

### 0.1 项目名称

```text
SCL_Channel_Coding_Design
```

### 0.2 本文档的作用

本文档用于规划三类信道编码仿真工程中“公共基础部分”的实现、测试、验证和阶段划分。

公共基础部分不直接实现 BCH、卷积码或 LDPC 的具体编译码算法，而是建立三类编码共同使用的基础能力，包括：

- 公共随机信息帧；
- 公共随机数和噪声机制；
- BPSK 调制；
- AWGN 信道；
- 硬判决解调；
- LLR 计算；
- 统一码率；
- BER、FER、成功率和时延统计；
- 停止规则；
- checkpoint/resume；
- CSV、JSON、日志和曲线输出；
- 公共仿真 runner；
- 面向 BCH、卷积码和 LDPC 的统一接口。

本文档同时规划后续多个 Common Stage，使每个阶段目标清晰、可单独测试、可单独审查，避免一次性编写大型程序后难以定位问题。

# 1. 项目总体目标

老师要求围绕三类编码开展仿真：

| 业务类型 | 输入电文长度 | 编码方案 |
|---|---:|---|
| 低速电文 | 约 200 bit、约 300 bit | BCH |
| 高速电文 | 约 300 bit | 卷积码、5G NR LDPC |

三类编码虽然具体算法不同，但基础仿真链路具有大量共同环节：

```text
原始信息帧
→ 码长适配
→ 信道编码
→ 可选交织
→ BPSK 调制
→ 信道传输
→ 硬判决或软信息计算
→ 可选去交织
→ 信道译码
→ 电文恢复
→ BER/FER/时延等统计
→ 结果保存与绘图
```

因此，应先建立稳定、正确、可复用的公共基础，再分别接入：

```text
BCH
→ 卷积码
→ LDPC
```

这样做的主要意义是：

1. 三类编码使用相同的码率定义；
2. 三类编码使用相同的 BPSK 映射；
3. 三类编码使用相同的 AWGN 与 LLR 公式；
4. 同类译码算法对比时能够共享同一帧、同一噪声和同一接收数据；
5. 不需要在 BCH、卷积码和 LDPC 目录中重复实现信道和统计代码；
6. 公共模块一旦验证正确，后续发现问题时可优先检查具体编译码算法；
7. 便于进行 C++、Python 和 MATLAB 三方验证；
8. 便于生成统一格式的结果和图片。

# 3. 统一定义

## 3.1 长度定义

后续三类编码中必须区分以下长度。

### 3.1.1 `K_payload`

原始输入电文长度。

例如：

```text
BCH 短电文：200 bit
BCH 中等电文：300 bit
卷积码高速电文：300 bit
LDPC 高速电文：300 bit
```

这是 BER 和 FER 统计的有效信息范围。

### 3.1.2 `K_codec_input`

真正进入编码器的信息长度。

它可能包含：

- 原始 payload；
- filler；
- CRC；
- 卷积码尾比特；
- 其他码长适配位。

例如，300 bit 卷积码整块零尾编码：

```text
K_payload = 300
tailLength = 6
K_codec_input = 306
```

### 3.1.3 `N_encoded`

编码或码率适配后的最终编码长度。

本项目统一码率使用：

\[
R=\frac{K_{\text{payload}}}{N_{\text{encoded}}}
\]

其中：

- 分子只使用原始输入信息长度；
- filler 不进入分子；
- CRC 不进入分子；
- 卷积码尾比特不进入分子；
- 分母使用最终编码后实际长度。

### 3.1.4 `N_transmitted`

实际送入 BPSK 调制和信道的比特数。

大多数实验中：

```text
N_transmitted = N_encoded
```

但仍需保留独立字段，防止后续打孔、缩短、占位 LLR 或特殊速率匹配造成定义混乱。

---

## 3.2 码率定义

本项目统一规定：

\[
R=
\frac{\text{原始输入长度}}
{\text{编码后实际长度}}
=
\frac{K_{\text{payload}}}{N_{\text{encoded}}}
\]

### 示例 1：BCH 整块 200 bit

```text
K_payload = 200
N_encoded = 248
R = 200 / 248 ≈ 0.80645
```

### 示例 2：BCH(15,11,1) 分块 200 bit

200 bit 按 11 bit 分组，需要：

\[
\left\lceil \frac{200}{11}\right\rceil=19
\]

组。

因此：

```text
fillerLength = 19×11 - 200 = 9
N_encoded = 19×15 = 285
R = 200 / 285 ≈ 0.70175
```

注意不能使用母码率 \(11/15\) 作为整帧有效码率。

### 示例 3：卷积码 300 bit、1/2 零尾

```text
K_payload = 300
tailLength = 6
K_codec_input = 306
N_encoded = 612
R = 300 / 612 ≈ 0.49020
```

### 示例 4：LDPC 300 bit、输出 576 bit

```text
K_payload = 300
N_encoded = 576
R = 300 / 576 ≈ 0.52083
```

---

## 3.3 BPSK 映射

统一规定：

\[
0\rightarrow +1
\]

\[
1\rightarrow -1
\]

例如：

```text
bit:     0   1   1   0
symbol: +1  -1  -1  +1
```

### 陌生词解释：BPSK

BPSK 是 Binary Phase Shift Keying，即二进制相移键控。

在本仿真中不必真正生成载波波形，只需把比特映射成实数符号：

```text
0 → +1
1 → -1
```

后续 AWGN 噪声直接加到这些实数符号上。

---

## 3.4 AWGN 信道

接收符号为：

\[
y_i=x_i+\sigma z_i
\]

其中：

- \(x_i\)：发送的 BPSK 符号；
- \(z_i\sim\mathcal N(0,1)\)：标准高斯随机数；
- \(\sigma\)：噪声标准差；
- \(y_i\)：接收符号。

统一使用：

\[
\sigma^2=
\frac{1}
{2R10^{E_b/N_0/10}}
\]

因此：

\[
\sigma=
\sqrt{
\frac{1}
{2R10^{E_b/N_0/10}}
}
\]

### 陌生词解释：AWGN

AWGN 是 Additive White Gaussian Noise，即加性白高斯噪声。

- “加性”表示噪声直接与信号相加；
- “白”表示不同采样点之间理想情况下互不相关；
- “高斯”表示噪声服从正态分布。

### 陌生词解释：\(E_b/N_0\)

\(E_b/N_0\) 表示每个信息比特能量与噪声功率谱密度之比，常用 dB 表示，是信道质量的重要横轴指标。

数值越大，通常表示信道越好、噪声越弱。

---

## 3.5 硬判决

统一规定：

```text
y >= 0 → bit 0
y < 0  → bit 1
```

必须明确 `y == 0` 时判为 0，避免不同模块自行决定边界行为。

### 陌生词解释：硬判决

硬判决只保留最终的 0/1 结果，不保留接收符号离判决边界有多远。

例如：

```text
y = +0.01 → 判为 0
y = +5.00 → 也判为 0
```

硬判决无法区分这两个结果的可靠程度。

---

## 3.6 LLR

统一定义：

\[
LLR_i=
\ln
\frac{P(c_i=0|y_i)}
{P(c_i=1|y_i)}
\]

对于本项目的 BPSK 和 AWGN：

\[
LLR_i=\frac{2y_i}{\sigma^2}
\]

符号含义：

```text
LLR > 0：更倾向 bit 0
LLR < 0：更倾向 bit 1
|LLR| 越大：判决置信度越高
```

### 陌生词解释：LLR

LLR 是 Log-Likelihood Ratio，即对数似然比。

它不仅表示判成 0 还是 1，还表示这个判断有多可靠。

例如：

```text
LLR = +0.1：稍微倾向 0
LLR = +10：强烈倾向 0
LLR = -0.2：稍微倾向 1
LLR = -12：强烈倾向 1
```

LLR 将用于：

- 卷积码软判决 Viterbi；
- LDPC Layered BP；
- LDPC Layered NMS。

# 4. 公共随机信息帧

## 4.1 帧类型

建立两类公共信息帧：

```text
K = 200 bit
K = 300 bit
```

使用范围：

| 编码类型 | 200 bit | 300 bit |
|---|---:|---:|
| BCH | 正式 | 正式 |
| 卷积码 | 不作为主场景 | 正式 |
| LDPC | 不作为主场景 | 正式 |

## 4.2 帧生成规则

每一帧按独立均匀二进制随机方式生成：

\[
P(0)=P(1)=0.5
\]

同一个 `frameIndex` 在不同编码方案中应对应相同的原始 payload。

例如：

```text
frameIndex = 0
BCH 使用 payload_0
卷积码使用同一个 payload_0 的前提是输入长度相同
LDPC 使用同一个 payload_0
```

## 4.3 帧池

公共帧池最大支持：

```text
50000 帧
```

建议按 shard 保存：

```text
frames_000000_009999.bin
frames_010000_019999.bin
frames_020000_029999.bin
frames_030000_039999.bin
frames_040000_049999.bin
```

### 陌生词解释：shard

shard 可理解为“分片文件”。

不是一次把 50000 帧全部写进一个大文件，而是每 10000 帧保存成一个独立文件。

优点：

- 单个文件较小；
- 便于校验；
- 便于追加；
- 某个分片损坏时不影响全部帧；
- smoke、prescan、formal 可以读取不同数量的分片。

## 4.4 manifest

每个帧池需要有 `manifest.json`。

### 陌生词解释：manifest

manifest 是“清单文件”，用于说明一组数据是如何生成的。

至少记录：

```text
framePoolId
payloadLength
totalFrames
shardSize
seed
generationAlgorithm
bitStorageFormat
endianness
createdTime
generatorVersion
每个 shard 的 SHA256
```

## 4.5 SHA256

### 陌生词解释：SHA256

SHA256 是一种文件指纹算法。

同一个文件内容会得到固定的哈希值；只要文件哪怕改变一个 bit，哈希通常都会明显变化。

它用于确认：

- 帧池是否被意外修改；
- C++、Python、MATLAB 是否读取同一个数据文件；
- checkpoint 恢复时帧池是否仍然一致。

# 5. 公共噪声设计

## 5.1 每帧独立随机噪声

老师要求：

```text
每一帧噪声必须随机生成
```

含义是：

- frame 0 与 frame 1 的噪声不能相同；
- 同一帧不同码元的噪声样本应独立；
- 不能只生成一条噪声并让所有帧重复使用；
- 固定 seed 后仍应能够复现。

## 5.2 `noiseGroupId`

不能简单用 decoder 名称生成噪声，否则硬判决和软判决会获得不同信道数据。

因此引入：

```text
noiseGroupId
```

### 陌生词解释：noiseGroupId

`noiseGroupId` 表示“公平对比组”。

同一个公平对比组中的不同算法共享相同噪声。

例如：

```text
卷积码硬判决 Viterbi
卷积码软判决 Viterbi
```

两者属于同一 `noiseGroupId`。

同理：

```text
LDPC Layered BP
LDPC Layered NMS
```

也共享同一 `noiseGroupId`。

## 5.3 seed 分层

建议定义：

```text
masterSeed
payloadSeed
noiseSeed
interleaverSeed
```

其中：

- `masterSeed`：整个实验的总随机种子；
- `payloadSeed`：信息帧生成；
- `noiseSeed`：信道噪声；
- `interleaverSeed`：伪随机交织顺序。

噪声上下文至少包含：

```text
masterSeed
noiseGroupId
frameIndex
```

## 5.4 是否跨 SNR 共享标准高斯母噪声

建议默认：

```text
reuseNoiseAcrossSnr = true
```

即同一帧在不同 SNR 下使用同一条标准高斯序列 \(z_i\)，只改变 \(\sigma\)。

```text
同一 frameIndex
同一 noiseGroupId
同一 z
不同 Eb/N0 → 不同 sigma
```

这样做的优点：

- 相邻 SNR 点更容易比较；
- 曲线波动更小；
- 更容易检查噪声缩放；
- 更容易保证算法公平。

注意，这不等于所有帧共享同一噪声。不同 frameIndex 仍使用不同噪声。

## 5.5 公共母噪声长度

初期建议：

```text
motherNoiseLength = 1000
```

因为老师要求单个编码块编码后总码长不超过 1000 bit。

各 case 根据实际发送长度取前 \(N_{\text{transmitted}}\) 个样本。

必须检查：

```text
N_transmitted <= motherNoiseLength
```

若不满足，应立即报错，不能循环使用或静默截断噪声。

## 5.6 跨语言一致性

直接依赖：

```text
C++ std::normal_distribution
Python numpy.random.normal
MATLAB randn
```

不能保证逐样本完全一致。

建议采用两层设计。

### 第一层：reference noise

用于 smoke 和跨语言验证，保存少量固定噪声文件。

```text
reference_noise.bin
reference_noise_manifest.json
```

C++、Python 和 MATLAB 读取同一个文件，逐样本比较。

### 第二层：formal noise

正式实验可采用统一、明确、版本冻结的随机数算法在线生成，但必须记录：

```text
PRNG 类型
正态变换方式
seed 派生方式
版本号
```

### 陌生词解释：PRNG

PRNG 是 Pseudo-Random Number Generator，即伪随机数生成器。

它不是物理真随机，但只要种子和算法固定，就能生成可复现的随机序列。

# 6. 公共统计指标

## 6.1 BER

\[
BER=
\frac{\text{原始信息错误 bit 数}}
{\text{原始信息总 bit 数}}
\]

只统计 payload。

不统计：

- filler；
- CRC；
- 卷积码尾比特；
- 校验位；
- LDPC 填充位。

### 陌生词解释：BER

BER 是 Bit Error Rate，即误比特率。

例如总共比较 100000 个 payload bit，其中 100 个错误：

\[
BER=100/100000=10^{-3}
\]

## 6.2 FER

一帧 payload 中只要有至少一个 bit 错误，就记为错误帧：

\[
FER=
\frac{\text{错误帧数}}
{\text{总帧数}}
\]

### 陌生词解释：FER

FER 是 Frame Error Rate，即误帧率。

例如发送 1000 帧，其中 20 帧至少存在一个错误 bit：

\[
FER=20/1000=0.02
\]

## 6.3 payload 成功率

\[
payloadSuccessRate=1-FER
\]

它表示最终恢复的原始电文完全正确的比例。

## 6.4 decoder 声明成功率

另行记录：

```text
decoderDeclaredSuccessRate
```

它表示译码器自身判断“译码成功”的比例。

两者不能混为一谈。

例如 LDPC 可能出现：

```text
syndrome = 0
但 payload 仍错误
```

这属于未检出错误。

## 6.5 未检出错误

```text
undetectedErrorFrames
```

定义：

```text
译码器声明成功
但 payload 实际错误
```

对应：

\[
undetectedErrorRate=
\frac{undetectedErrorFrames}{totalFrames}
\]

## 6.6 时延

至少统计：

```text
avgEncodeTime_us
maxEncodeTime_us
avgDecodeTime_us
maxDecodeTime_us
```

可扩展：

```text
p95DecodeTime_us
interleaveTime_us
deinterleaveTime_us
endToEndAlgorithmTime_us
```

### 陌生词解释：P95

P95 表示 95% 的样本都不超过该时延。

例如：

```text
P95 decode time = 80 μs
```

表示 95% 的译码时延不超过 80 微秒。

## 6.7 计时范围

算法计时不包含：

```text
文件 IO
日志打印
CSV 写入
JSON 写入
图片绘制
帧池读取
噪声文件读取
```

短算法应支持 batch timing。

### 陌生词解释：batch timing

batch timing 表示把同一个短操作连续执行很多次，再用总时间除以次数。

这样可以降低计时器分辨率和系统抖动的影响。

## 6.8 迭代次数

LDPC 等迭代译码需要记录：

```text
avgIterations
maxIterations
earlyStopFrames
maxIterationFrames
```

公共模块只提供统计接口，不实现 BP/NMS。

## 6.9 置信区间

BER 和 FER 是有限样本估计值，应输出 95% 置信区间。

字段：

```text
berCiLower
berCiUpper
ferCiLower
ferCiUpper
confidenceLevel = 0.95
```

### 陌生词解释：置信区间

置信区间表示根据有限实验样本，对真实 BER/FER 可能范围的统计估计。

它不是说“真实值有 95% 概率在这里”，而是说如果重复采用同样方法构造区间，大约 95% 的区间会覆盖真实值。

## 6.10 零错误点

如果运行 50000 帧后 `frameErrors = 0`，CSV 中真实 FER 仍写为 0。

但在对数曲线上不能直接画 0，因此需单独生成：

```text
plotFER
isZeroErrorPoint
```

可使用“三倍法则”给出 95% 上界近似：

\[
FER_{\text{upper}}\approx\frac{3}{N_{\text{frames}}}
\]

例如：

```text
N = 50000
FER upper ≈ 6×10^-5
```

图片中必须说明零错误点的显示方式。

# 7. 编码增益

编码增益建议在固定 BER 或 FER 目标点计算：

\[
G_{\text{coding}}
=
(E_b/N_0)_{\text{reference}}
-
(E_b/N_0)_{\text{candidate}}
\]

参考曲线优先使用：

```text
uncoded BPSK
```

例如，在：

```text
FER = 10^-2
```

处：

```text
未编码 BPSK 需要 6.0 dB
某编码方案需要 3.5 dB
编码增益 = 2.5 dB
```

编码增益属于曲线级指标，不放在每个 SNR 点的行中。

结果拆分为：

```text
point_results.csv
curve_summary.csv
```

# 8. 公共停止规则

## 8.1 smoke

建议：

```text
minFrames = 20
targetFrameErrors = 5
maxFrames = 100
```

目的：

- 检查程序可运行；
- 检查无噪声链路；
- 检查每帧噪声不同；
- 检查 BER/FER 统计；
- 检查 CSV、日志、trace 和图片；
- 快速暴露数组越界、长度错误和符号错误。

### 陌生词解释：smoke

smoke test 即冒烟测试。

它不是正式性能实验，而是用很少的数据确认系统基本“没有冒烟、没有立刻坏掉”。

## 8.2 prescan

建议：

```text
minFrames = 300
targetFrameErrors = 30
maxFrames = 2000
SNR step ≈ 0.5 dB
```

目的：

- 粗略找到 waterfall 区域；
- 确定 formal 的 SNR 范围；
- 估算运行时间；
- 检查曲线方向和数量级。

### 陌生词解释：waterfall

waterfall 表示编码曲线上 BER/FER 随 SNR 增加而快速下降的区域，形状像瀑布。

## 8.3 formal trial

建议：

```text
minFrames = 1000～5000
targetFrameErrors = 50～100
maxFrames = 5000～20000
```

目的：

- 冻结正式实验参数；
- 检查运行速度；
- 检查 checkpoint/resume；
- 确定 formal 预计总时长；
- 确认结果文件规模。

## 8.4 formal

初步建议：

```text
minFrames = 5000
targetFrameErrors = 200
maxFrames = 50000
SNR step = 0.1～0.2 dB
```

停止条件：

```text
(frames >= minFrames AND frameErrors >= targetFrameErrors)
OR
frames >= maxFrames
```

停止原因统一为：

```text
TARGET_ERRORS_REACHED
MAX_FRAMES_REACHED
MANUAL_STOP
ERROR_ABORT
```

# 9. checkpoint/resume

## 9.1 含义

### checkpoint

checkpoint 是运行中的保存点。

程序运行时间较长时，定期保存当前进度。

### resume

resume 是从 checkpoint 恢复运行。

恢复后不能从头重新生成不同的 payload 或噪声，而应继续得到与连续运行一致的结果。

## 9.2 checkpoint 内容

至少保存：

```text
stageId
runId
caseId
snrIndex
nextFrameIndex
framesProcessed
bitErrors
frameErrors
payloadSuccessFrames
decoderDeclaredSuccessFrames
undetectedErrorFrames
timingAccumulator
iterationAccumulator
configHash
framePoolHash
noisePolicyVersion
codeVersion
gitCommit
```

## 9.3 恢复前检查

恢复时必须验证：

```text
当前配置 hash == checkpoint 配置 hash
当前帧池 hash == checkpoint 帧池 hash
当前代码版本满足恢复要求
当前 caseId 和 SNR 一致
```

不一致时禁止继续。

## 9.4 验证目标

必须证明：

```text
连续运行 1000 帧
```

与：

```text
先运行 400 帧
保存 checkpoint
恢复后继续 600 帧
```

最终结果完全一致。

# 13. 公共 runner

公共 runner 只负责组织流程：

```text
读取 payload
→ 编码
→ 交织
→ BPSK
→ 信道
→ 硬判决和/或 LLR
→ 去交织
→ 译码
→ payload 恢复
→ 指标统计
→ 停止判断
→ checkpoint
→ 输出
```

runner 禁止包含：

```text
BCH 多项式
Viterbi ACS
LDPC 节点更新
某个具体码型的硬编码判断
某个具体 decoder 的专用逻辑
```

具体编码器和译码器通过接口注入。

### 陌生词解释：接口注入

接口注入表示 runner 只知道“这是一个编码器”或“这是一个译码器”，不关心内部具体算法。

例如后续可以把：

```text
BCHEncoder
ConvolutionalEncoder
LDPCEncoder
```

分别传给同一个 runner。

Q 函数表示标准正态随机变量超过某个阈值的概率。

# 20. 总结

公共基础仿真不是简单地写一个 AWGN 程序，而是建立整个三类编码实验的统一底座。

最终公共基础应具备：

```text
统一数学定义
统一数据
统一噪声
统一信道
统一指标
统一停止规则
统一输出
统一恢复机制
统一接口
统一验证证据
```

公共基础通过后，后续 BCH、卷积码和 LDPC 的开发重点才能集中到各自的编译码算法，而不是反复修改帧、噪声、码率、统计和绘图逻辑。
